# AUT-001 — gate controlado e validação contínua

**Data:** 2026-03-23  
**Escopo:** fase 2 da entrega de liveness / auto-close auditável

---

## 1. Gate controlado

### Objetivo

Permitir que o `omega` transite sessões de `stale` para `closed` apenas depois de validação manual, preservando a proteção das sessões em uso real.

### Procedimento

1. rodar `python scripts/session_liveness_guard.py --report-path reports/liveness-report-YYYYmmddTHHMMSSZ.json` (o runner faz isso automaticamente)
2. revisar o JSON gerado, olhando os campos:
   - `decision`
   - `reason`
   - `minutes_since_signal`
   - `active_protections`
3. se o candidato for `auto_close_candidate` e não houver sinais de proteção, executar:
   - `python scripts/session_liveness_guard.py --apply --report-path reports/liveness-apply-...json`
4. registrar no `connectome/status.json` que a decisão foi tomada, atualizando:
   - `liveness.last_decision`
   - `closed_at`
   - `closure_reason`
5. documentar a decisão em `knowledge/` (ex.: `knowledge/2026-03-23-aut001-gate.md`)

## 2. Validação / testes

Para validar o fluxo sem encerrar sessão real:

- criar uma cópia local da sessão em `sessions/test-*.json` com tempo de `last_activity_at` antigo
- rodar `python scripts/session_liveness_guard.py --sessions-dir sessions/test --apply --report-path reports/test-report.json`
- confirmar que a sessão de teste recebe `close_status` e `closure_reason` corretamente
- garantir que a sessão principal continua protegida por `active_protections`
